Android Handler机制 -- 源码分析

Posted on 2021-06-22 Edited on 2021-06-23 In Android学习笔记

为什么要使用Handler机制？

在Android开发中，为了保证UI操作的线程安全性，UI组件的更新变化只允许在UI线程、即主线程上进行。
但是在实际开发中经常会有 其他线程需要修改UI组件的情况，所以有了Handler机制来帮助传递 更新UI的需求信息。
将工作线程需操作UI的消息（Message） 传递 到主线程，使得主线程可根据工作线程的需求 更新UI，从而避免多个线程并发操作UI导致线程操作不安全的问题

Handler机制 继而可以应用于其他线程通信的需求，负责整个App中信息的传递

运作方式

Handler机制主要包含四个要素：Handler、Looper、MessageQueue、Message/Runnable

Looper（循环器）是整个机制的核心，相当于是消息泵、或消息引擎。在Looper运转的过程中，其自身机制会保证不断的从MessageQueue中获取下一个信息（Message）并执行

MessageQueue（消息队列）是按顺序存放Message的数据结构，主要特性和Queue一样，先入先出

Handler（处理器）提供了管理Looper、调用Looper、以及和外部组件交互的能力

Handler 初始化

官方文档：
A Handler allows you to send and process Message and Runnable objects associated with a thread’s
MessageQueue. Each Handler instance is associated with a single thread and that thread’s
message queue. When you create a new Handler, it is bound to the thread message queue
of the thread that is creating it — from that point on, it will deliver messages and
runnables to that message queue and execute them as they come out of the message queue.

There are two main uses for a Handler: (1) to schedule messages and runnables to be
executed as some point in the future; and (2) to enqueue an action to be performed on
a different thread than your own

翻译过来主要含义就是：
每个Handler要和一个thread和该thread对应的MessageQueue绑定，用于处理该消息队列上的Message/Runnable对象。
所以，一个新的handler会和他被创建时所在的线程、以及该线程的MessageQueue绑定，他会把Message/Runnable入队到
该线程的MessageQueue上，并在这个Message/Runnable出队时进行处理

Handler的两个主要用途为：（1）调度 messages和runnables，是他们可以在未来某时刻被处理（异步处理）
（2）把某个action入队，使其可以在其他线程上被执行

Looper是消息引擎，在需要与外界交互的时候必传
目前默认无参的构造器 Handler() 已经 @Deprecated，官方注释是为了避免一些隐式的bug出现（比如Looper inactive、线程绑定错误等）。
官方建议使用Executor，或显式指定Looper的方式来构造Handler

执行的优先级排序从大到小为：message.callback > handler.callback > handler.handleMassage()

async（Boolean）参数，表示是否异步，默认值为false。一般情况下手动初始化Handler时不需要传入这个参数，

在Handler初始化的时候，该handler实例会和指定的Looper，以及该Looper绑定的MessageQueue绑定上。无论sendMessage/post
发生在哪个线程上，都会在已绑定的Looper和MessageQueue所在的线程上执行信息处理操作。

三种消息类型

在Handler入队消息时，会先检查mAsynchronous，也就是初始化时async输入的值。True则该msg需要异步执行

由于每一个线程，只有一个对应的消息队列MessageQueue、以及一个获取消息的Looper，但是一个线程可以有很多个Handler负责处理不同的消息。

Looper

Looper对象的创建主要通过Looper.prepare()（当前线程）/Looper.prepareMainLooper()（UI线程）
MessageQueue是Looper的成员变量(mQueue)，在Looper.prepare()的阶段会初始化Looper，并准备好消息队列

同步屏障（SyncBarrier）

MessageQueue对于异步消息是如何支持的？SyncBarrier就是有效保证异步消息正常执行的机制。由于三种类型的信息都在
同一个MessageQueue中，因此当同步屏障按规则插入队列后，Looper在执行next()到下一个时若遇到一个同步屏障，
则会直接寻找后续队列中的第一个异步Message，并拿出来执行。

以下为官方文档：
Message processing occurs as usual until the message queue encounters the
synchronization barrier that has been posted. When the barrier is encountered,
later synchronous messages in the queue are stalled (prevented from being executed)
until the barrier is released by calling MessageQueue.removeSyncBarrier() and specifying
the token that identifies the synchronization barrier.

postSyncBarrier() method is used to immediately postpone execution of all subsequently posted
synchronous messages until a condition is met that releases the barrier. Asynchronous
messages are exempt from the barrier and continue to be processed as usual.

A call to post method must be always matched by a call to removeSyncBarrier() with the
same token to ensure that the message queue resumes normal operation. Otherwise the
application will probably hang!

从代码中可以看出，同步屏障信息 实际上是直接从全局的信息池中取出来的instance，并没有target到某个Handler。

总结

  • 一个线程只能有一个Looper和一个MessageQueue
  • 一个线程可以有多个Handler
  • 主线程（UI线程）的Looper & MessageQueue 会在主线程创建时默认创建，直接使用即可；子线程则需要手动创建
  • Runnable对象通过 post()入队，callback方法处理； Message通过sendMessage()入队，handleMessage()处理
  • 异步信息可以越过同步屏障，不会被阻塞

Groovy 简介

Gradle脚本使用的是Groovy语言，Groovy也是基于JVM的一种动态语言，基于Java并扩展了Java语言。Groovy会使写Java程序就要写脚本一样简单，写完就可以执行，所以看起来也像是一门脚本语言。Groovy内部会把代码编译为Java class文件然后启动虚拟机来执行，这些过程，我们开发人员是不用管的。

Gradle是一种DSL，也即是 Domain Specific Language 领域特定语言。说白了DSL就是一种行话，这个语言在Gradle中可以识别，在其它地方就用不了了。

学习Gradle，我们不必要精通Groovy语言，但还是需要了解一些Groovy的基本知识，比如def关键字、返回语句、字符串、集合、闭包等。下面，我们就来简单介绍一下。

Groovy中的字符串

Groovy支持多种字符串：单引号、双引号、三引号
单引号会直接打印字符串的内容，不会对’$’标注的内容进行转义
双引号可以使用’$’对变量 或 表达式进行转义
三引号可以支持字符串换行、保留字符串内的所有格式（和kotlin中的三引号功能类似）

  • Groovy默认把 函数最后一行的执行结果 作为该函数的返回值，如果所需要的结果正好在最后一行、则可以省略return
  • Groovy和很多动态语言一样，”;”并不是必要的，
  • 在Groovy中调用函数可以不加括号

def关键字

在定义变量或者方法的时候使用def。如果没有指定具体的参数类型，Groovy在运行时会自动判断类型。（def也可以省略，效果一样）


使用def定义方法的返回值类型，可以返回任意类型。当然如果指定了具体的返回值类型，就必须正确返回该类型的值

Posted on 2021-05-21 Edited on 2021-06-23 In Kotlin学习笔记

I.协程的基本概念

协程是在语言上实现的比线程更小的单位，可以看做为轻量级的线程。但协程依然运行在线程上，是用户态、完全由程序所控制的，也因此不会像线程切换那样消耗资源。
在内核上最小的调度单位依然是线程，一个线程上可以有多个协程。但是一个线程上的多个协程一定是串行运行的，当一个协程执行时，其他协程必须挂起

  • 协程是可以由程序自行控制挂起、恢复的程序/函数
  • 协程可以用来实现多任务的协作执行
  • 协程可以用来解决异步任务控制流的灵活转移
  • 支持阻塞线程的协程和不阻塞线程的协程

协程的操作主要就是：挂起 & 恢复两个操作

1. 作用

协程可以让异步逻辑的代码同步化
降低异步程序的设计复杂度，可以让并发任务更加轻量
协程主要在控制代码的流程，让代码更简单。并不会主动帮我们实现异步

  • 挂起和恢复可以控制执行流程的转移
  • 异步逻辑可以用同步代码的形式写出
  • 同步代码比异步代码更灵活，更容易实现复杂业务

2. 协程的异步机制

非阻塞式的挂起：主调用流程挂起当前执行的协程，把调用流程切到子协程上，主协程等待回调时再恢复执行

3. 线程 vs 协程

线程Thread：指操作系统的线程，也称为内核线程
协程Coroutine：指语言实现的协程，运行在内核线程之上

4. kotlin协程支持

语言级别（协程要素）：kotlin标准库 kotlin.coroutines.*（协程上下文，拦截器，挂起函数）

框架级别的支持：官方协程框架 kotlinx.coroutines.*（Job，调度器，作用域）

kotlin协程学习步骤： 协程的概念 -> 协程要素的运用 -> 协程框架的运用

II. 协程的常见实现

1. 协程的分类

  • 按调用栈

    • 有栈协程：每个协程会分配单独的调用栈，类似线程的调用栈，通过栈来保存调用对象
      可以在任意函数嵌套中挂起，例如 Lua Coroutine
    • 无栈协程：不会分配单独的调用栈，挂起点状态通过闭包或对象保存（Kotlin， JavaScript， C#）
      无栈协程的挂起 和 恢复 不能有函数的嵌套，只能在当前函数中挂起 (eg: Python Generator)
      如今大部分是无栈，可以节省内存资源

  • 按调用关系

    • 对称协程：调度权可以转移给任意协程，协程之间是对等关系
    • 非对称协程：调度权只能转移给调用自己的协程，协程存在父子关系（大部分都是非对称），调用其他协程后
      还必须回到自身协程继续后面的逻辑，即只能由父协程调用子协程并获取返回值，
      yield只能将调度权转移给对应的resume

2. 常见实现

协程主要的操作：挂起 & 恢复
不同的语言形成了不同的实现方式：

Python Generator：无栈的非对称协程

大体的运行流程为：

  1. gen = numbers()， 返回一个Generator
  2. print(f”[0] {next(gen)}”)，next(gen)进入numbers()函数，主协程被挂起，调起一个子协程执行numbers()方法
  3. numbers()一直执行到 yield i，子协程让步挂起，返回此时 i=0 的值给主协程
  4. 主协程得到子协程的结果，恢复执行，print（f”[0] {0}”）
  5. 主协程执行到下一个 next(gen)，又进入到子协程中，主协程被挂起
  6. 子协程恢复，从yield i位置继续向下执行
  7. 子协程经过while进入下一个循环，执行到yield的时候，再次挂起并返回此时 i=1的值
  8. 主协程接收到i=1的值，恢复执行 print(f”[1] {1}””)

Lua Coroutine：有栈的非对称协程

有栈：函数可以嵌套调用多个协程，可以任意挂起和恢复
创建协程：coroutine.create()，返回了一个协程对象
查询状态：coroutine.status()
挂起协程：coroutine.yield()
恢复协程：（要从其他协程中调用）coroutine.resume()

  1. main中执行两个协程的创建，进入repeat循环
  2. 主协程执行到coroutine.resume(producerCoroutine)被挂起，进入子协程执行producer()
  3. 子协程中执行producer，进入for循环，打印出”send 0”
  4. 子协程执行到coroutine.yield(i), 把i=0的值返回，子协程挂起
  5. 主协程获得返回值，并恢复执行，(true, 0) = coroutine.resume(producerCoroutine), (status为true是因为子协程之间还没有执行完)
  6. 主协程执行到coroutine.resume(consumerCoroutine, 0) 被挂起，进入另一个子协程执行consumer()
  7. 子协程 print(“receive 0)
  8. 子协程 执行yield()挂起，等待下一次的resume调用接收value值
  9. 主协程 恢复 此时status依然为true，进入下一次循环，知道producer协程结束

Go routine: 对称协程（相互之间独立，可以同时开始）

  1. 两个协程并发，可以同时开始执行
  2. read协程执行到 readChannel 时，由于此时channel中没有值 -> read协程block被挂起（转移调度权）
  3. write协程执行到 writeChannel “0”挂起 等待int被取出
  4. read协程发现channel有值自动恢复，打印”read 0”， write协程发现值被取出也自动恢复，两个协程并发执行

对称协程：调度权转移后不用回到原来的协程
channel 任意协程都可以读写

  • 每个go routine都是并发或并行执行（协程可以调度到线程上，线程支持多核 -> 即并发）
  • 无buffer的channel写时会挂起，直到读取，反之亦然
  • go routine 可以认为是一种有栈对称协程的实现

async/await实现： JavaScript, C#, Python, Rust… (kotlin也可以实现)

  • 可以多层嵌套，但必须为async function （两种函数：function 和 async function)
  • async/await 是一种无栈非对称的协程实现（不能在任意函数中随意嵌套，只能在async中，所以为无栈）
  • 目前各类语言支持最为广泛的特性

基于非对称协程实现对称协程：
对称协程：协程A —transfer(coroutineB)—> 协程B
非对称协程：协程A —yield()—> 调度中心 —resume(coroutineB)—> 协程B
调度中心可以是另一个协程、也可以是其他任何形式的调度器、中间件
基本上任何问题都可以通过增加一个间接的中间层来解决

3. kotlin协程的基本要素

Kotlin标准库中提供的最基本的元素

挂起函数 suspend

suspend关键字 + 函数/lambda表达式 -> 该函数即为挂起函数
挂起函数的调用处称为：挂起点

  • 挂起函数：以suspend修饰的函数
  • 挂起函数只能在其他挂起函数或协程中调用 （所有function都可以在suspend function中调用，
    但suspend function不能在其他function中调用）（kotlin是无栈协程）
  • 挂起函数调用时包含了协程 “挂起” 的语义，即调用挂起函数时主调用流程被挂起
  • 挂起函数返回是则包含了协程 “恢复” 的语义

Continuation

无栈协程通过 闭包/对象 保存挂起点的状态，在Kotlin中使用continuation来保存

挂起函数的类型

如何做到挂起、恢复

编译器会帮我们在挂起函数里传入参数：continuation: Continuation<T>，
所以挂起函数只能在其他挂起函数中或协程中调用（只有协程 和 挂起函数 才有相应的continuation可以传入）

  • suspend关键字添加continuation参数
  • continuation中的泛型参数类型 是由挂起函数的返回类型决定的
  • Any:
    • 没有真正挂起时，Any承载函数真正返回的结果
    • 真正挂起时，函数通过回调来执行，Any承载的是一个挂起的标志（挂起标志对象：COROUTINE_SUSPENDED），告诉外部流程或协程体，该函数正在被挂起等待回调

将回调方式转写成挂起函数

  • 主要是通过suspendCoroutine函数的调用，获取当前挂起函数的Continuation
  • 回调成功的分支使用Continuation.resume(value)
  • 回调失败则使用Continuation.resumeWithException(e)

真正的挂起必须异步调用resume，包括：

  1. 切换到其他线程resume
  2. 单线程事件循环异步执行
    注意：直接return、没有切线程、在suspendCoroutine中直接调用resume也算没有挂起

协程的创建

  • 协程是一段可执行的程序
  • 创建需要一个函数： suspend function
  • 创建需要一个API：createCoroutine()， startCoroutine()

一个协程创建后会有两个continuation：

  • 传入的completion：Continuation<T>： 协程执行完之后回调的continuation。suspend函数本身执行需要一个Continuation实例在恢复时调用
  • 返回值Continuation：是创建出来的协程的载体, receiver suspend函数会被传给该实例作为协程的实际执行体（执行完他的resume后才会执行回调的continuation）
    协程本身也是一个continuation

suspend{}函数整体 会被包装成一个Continuation， 即为createCoroutine返回的协程载体的continuation -> 可以理解为协程本体，后面的操作也都在该continuation上进行
object：Continuation 回调逻辑也是一个Continuation

启动协程

创建并启动

协程中如果有n个挂起点，总共调用多少次resume？

  • 第一次启动协程，为suspend{}包装成的协程本身resume()一次
  • 协程执行结束，回调continuation会resume()一次
  • 如果有n个真正的挂起点（切换线程的挂起点），一共n+2次

协程上下文

val context = EmptyCoroutineContext
本质上为协程 数据的包装，Persistent context for the coroutine. It is an indexed set of [Element] instances.

  • An indexed set is a mix between a set and a map.
  • 协程执行过程中需要携带数据
  • 索引是CoroutineContext.Key
  • 元素是CoroutineContext.Element

拦截器

  • ContinuationInterceptor是一类协程上下文元素
  • 可以对协程上下文所在协程的Continuation进行拦截

Continuation 执行流程

SafeContinuation的作用就是确保：

  1. 包装的SuspendLambda resume只被调用一次
  2. 如果在当前线程调用栈上直接调用，则不会挂起，直接return（确保一定为真正的挂起）

拦截Continuation：
SuspendLambda实例 ➡️ Intercepted包装成一个新的Continuation ➡️ SafeContinuation包装成SafeContinuation实例
SafeContinuation.resume ➡️ 拦截器返回的Continuation.resume ➡️ 真正的函数体SuspendLambda.resume
在拦截器上执行切换线程，真正函数体在切换的线程上执行（拦截器最重要的作用之一是切换线程）

注意：

  • SafeContinuation仅在挂起点时出现
  • 拦截器在每次（恢复）执行协程体时调用
  • SuspendLambda是协程函数体

协程挂起恢复要点：

  • 协程体内的代码都是通过Continuation.resumeWith调用
  • 每调用一次continuation.resumeWith 的 label加1，每一个挂起点对应与一个case分支
  • 挂起函数在返回COROUTINE_SUSPENDED时才会挂起
    （如果返回的不是COROUTINE_SUSPENDED，则没有真正挂起，返回的是真正的结果）

协程的线程调度

主要是通过interceptor来实现：在interceptor里面将原始的Continuation替换成一个新的continuation
在interceptor里面先切线程，再执行SuspendLambda的逻辑，就完成了线程切换

Kotlin 学习笔记（8）-- 注解

I.注解的基本概念

  • 注解是对程序的附加信息说明
  • 注解可以对类、函数、函数参数、属性等做标注
  • 注解的信息可用于源码级、编译期、运行时
    • 源码期：生成文档时，给开发者看的
    • 编译期：给编译器看的，仅保留到编译期，运行时获取不到
    • 运行时：一直保留到运行时段，用于反射

II.定义注解类

SOURCE < BINARY < RUNTIME (可作用范围，RUNTIME会一直保留到代码运行)

注解输入的参数仅支持以下类型及其数组：基本类型， KClass，枚举，其他注解
要求是那些在编译期就能确定值的类型

III. 使用

@JvmName：指定 类/方法 在JVM中的名字, 让编译生成的类名 可以与 代码中的名字 不同

IV. 注解 vs 注释

Annotations: 参与编译的特定语法现象
Comments: 仅存在于源码中，提升可读性

V.常见内置注解

1. 用于标注注解的注解

import path: kotlin.annotation.*

2. 标准库的通用注解

import path: kotlin.*

  • Metadata：kotlin反射的信息通过该注解附带在元素上
  • UnsafeVariance：泛型用来破除型变限制
  • Suppress：用来去除编译器警告，警告类型作为参数传入

还有很多注解，可以去看documentation

3. java虚拟机相关注解

kotlin.jvm.* 用于与java虚拟机交互的注解

很重要，尤其现在很多项目都是 Kotlin 与 Java 一起使用，这些Jvm注解可以帮助我们在两种语言中达成一样的效果，进而避免一些可能引起的问题。

  • JvmField：生成java中的 field（kotlin中的属性可能到java中会生成为private，添加JvmField就是public了）
  • JvmName：给方法、文件、类等指定在jvm中的名字
  • JvmOverloads：函数默认参数生成函数重载。指定默认参数的方法，在java看来是一个overload的函数
  • JvmMultifileClass: 该jvm类有多个文件，会生成多个该名字的类，编译后其实都在一个文件上
  • JvmStatic：把方法编译成静态，生成静态成员
  • Synchronized：标记函数为同步函数
  • Throws：标记函数抛出的异常类型。kotlin没有受检异常，加在方法上的@Throws(IOException::class) 和java中 xxFunction throws IOException 一样
  • Volatile：生成volatile的 Field
  • Transient：配合序列化，标记属性不被序列化

这些都是常用的jvm注解，其他也可以到documentation中查看

2. 用途

  • 拿到类型的：属性、方法、内部类
  • 调用指定名称/签名的方法，或访问指定名称的属性（只要有类名 —> 能拿到类 —> 拿到类的属性、方法等等）
  • 通过签名信息获取泛型实参的具体类型（编译之后，泛型参数是要被擦除掉的，字节码中只有原始类型）
  • 访问运行时注解及其信息完成注入或者配置操作

3. 反射常用数据结构

数据结构 Java中对应结构 说明
KType Type 描述未擦除的类型或泛型参数，eg：Map。可通过typeOf或以下类型获取对应的父类、属性、函数参数等
KClass Class 描述对象的实际类型，不包含泛型参数，eg：Map；可通过对象、类型名直接获得
KProperty Field 描述属性，可通过属性引用、属性所在类的KClass获取
KFunction Method 描述函数，可通过函数引用、函数所在类的KClass获取

KType & KClass:

KType描述的就是我们直接使用的类型(Map)；KClass描述的是真实的类型(Map)，泛型擦除后的实际类型

KProperty & Field：

KProperty 比 Field 更强大。Kotlin中 property = field + setter + getter

KFunction & Method：

KFunction 既可以是顶级函数、也可以是类的方法函数。但Java中没有顶级函数概念，Method仅指类的方法

4. 使用反射

Java反射：

  • Pros：在Kotlin中可以直接使用、并且无需引入额外依赖
  • Cons：无法使用kotlin的语法特性，需对kotlin生成的字节码足够了解

    Kotlin反射：

  • Pros: 支持Kotlin中的几乎所有特性，API也更友好
  • Cons: 需要添加 kotlin反射库 的引用，首次调用慢

kotlin的反射信息是写在@Metadata里面的，所以首次调用有一个获取注解、反序列化信息的过程
